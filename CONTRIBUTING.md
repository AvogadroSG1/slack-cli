## Development Setup

### Prerequisites

| Tool | Version | Install |
|------|---------|---------|
| Go | 1.26+ | [go.dev/dl](https://go.dev/dl/) |
| golangci-lint | latest | `go install github.com/golangci/golangci-lint/cmd/golangci-lint@latest` |
| make | any | Pre-installed on macOS/Linux |

### First Time Setup

```bash
# Clone and build
git clone https://github.com/poconnor/slack-cli.git
cd slack-cli
make build

# Verify
./bin/slack-cli version
./bin/slack-cli api list --pretty

# Set your token for testing live API calls
export SLACK_TOKEN="<your-slack-bot-or-user-token>"
./bin/slack-cli auth test
```

### Environment Variables

| Variable | Required | Description |
|----------|----------|-------------|
| `SLACK_TOKEN` | Yes (for API calls) | Slack Bot or User token. Not needed for `make test` or `api list`. |

## How to Implement a Slack API Method

This is the most common contribution task. There are 73 methods in the registry, but many dispatch functions are stubs that return "not yet implemented." Implementing a method means writing the real function that calls the Slack SDK.

### Understanding the Dispatch Pattern

When the code generator runs, it creates two things per method:

1. A **registry entry** in `internal/registry/generated.go` that describes the method, its parameters, and pagination behavior. You do not edit this file.
2. A **stub dispatch function** in `internal/dispatch/generated_dispatch.go` that returns an error. You do not edit this file either.

Your job is to write a **real dispatch function** in the appropriate `impl_*.go` file. Because Go processes `init()` functions in filename-alphabetical order within a package, your `impl_chat.go` (which starts with `i`) runs after `generated_dispatch.go` (which starts with `g`). Your `RegisterDispatch` call overwrites the stub.

### Step by Step

Let us say you want to implement `reactions.add`. Here is the process:

**1. Check the registry entry.**

Look at what the registry says about this method. Find its entry in `internal/registry/generated.go`:

```go
{
    APIMethod:   "reactions.add",
    Category:    "reactions",
    Command:     "add",
    SDKMethod:   "AddReactionContext",
    CallStyle:   "positional",
    Params: []ParamDef{
        {Name: "name", SDKName: "name", Type: "string"},
        {Name: "item", SDKName: "item", Type: "json"},
    },
},
```

This tells you:
- The SDK method is `AddReactionContext` on `*slack.Client`
- It takes positional parameters: a reaction name and an item reference
- It is not paginated

**2. Check the Slack API docs.**

Open the `DocsURL` for the method: https://api.slack.com/methods/reactions.add

Understand what parameters it expects and what it returns. Cross-reference with the SDK method signature in the `slack-go/slack` package.

**3. Write the dispatch function.**

Open or create `internal/dispatch/impl_reactions.go`. Follow the existing pattern:

```go
package dispatch

import (
    "context"
    "fmt"

    "github.com/slack-go/slack"
)

func implReactionsAdd(ctx context.Context, client *slack.Client, flags map[string]any) (any, error) {
    name := flagStr(flags, "name")
    if name == "" {
        return nil, fmt.Errorf("name flag is required")
    }
    channel := flagStr(flags, "channel")
    if channel == "" {
        return nil, fmt.Errorf("channel flag is required")
    }
    ts := flagStr(flags, "ts")
    if ts == "" {
        return nil, fmt.Errorf("ts flag is required")
    }

    ref := slack.ItemRef{
        Channel:   channel,
        Timestamp: ts,
    }

    err := client.AddReactionContext(ctx, name, ref)
    if err != nil {
        return nil, fmt.Errorf("reactions.add: %v", err)
    }

    return map[string]any{"ok": true}, nil
}

func init() {
    RegisterDispatch("reactions.add", implReactionsAdd)
}
```

Key things to note:

- **Function signature**: Must match `DispatchFunc`: `func(ctx context.Context, client *slack.Client, flags map[string]any) (any, error)`
- **Flag extraction**: Use the helpers in `flags.go`: `flagStr`, `flagInt`, `flagBool`, `flagStrSlice`
- **Required flags**: Validate them and return clear error messages
- **SDK call**: Always use the `*Context` variant (e.g., `AddReactionContext`, not `AddReaction`) so context cancellation works
- **Return value**: Return a `map[string]any` or a struct. The output layer handles JSON serialization.
- **Registration**: The `init()` block registers your function, overwriting the generated stub

**4. Write tests.**

Add a test in `internal/dispatch/` following the table-driven pattern. See the [Testing](#testing) section below.

**5. Verify.**

```bash
make test
make build
./bin/slack-cli reactions add --help  # Should show your flags
```

## How to Add a New Slack API Method

This is different from implementing an existing method. If the Slack SDK supports a method that is not in our registry at all, you need to add it to the mapping table.

**1. Find the SDK method.**

Look in the `slack-go/slack` package for a method on `*Client` ending in `Context`. For example, `client.SetUserCustomStatusContext(...)`.

**2. Add it to the mapping table.**

Edit `cmd/introspect/mapping.go` and add an entry:

```go
var methodMapping = map[string]string{
    // ... existing entries ...

    // users.profile
    "SetUserCustomStatus": "users.profile.set",
}
```

The key is the SDK method name without the `Context` suffix. The value is the Slack API method name.

**3. Regenerate.**

```bash
make generate
```

This updates `internal/registry/generated.go` (new `MethodDef` entry) and `internal/dispatch/generated_dispatch.go` (new stub function).

**4. Implement the dispatch function.**

Follow the [How to Implement a Slack API Method](#how-to-implement-a-slack-api-method) walkthrough above.

## How to Override a Generated Command

Sometimes a method needs a completely custom Cobra command -- different flag parsing, special output formatting, or behavior that does not fit the generic dispatch pipeline. The `api list` command is an example.

**1. Write a custom Cobra command.**

Create a file in `internal/override/` (or add to an existing one):

```go
package override

import "github.com/spf13/cobra"

func init() {
    Register("some.method", &cobra.Command{
        Use:   "custom-name",
        Short: "Does something custom",
        RunE: func(cmd *cobra.Command, args []string) error {
            // Your custom implementation
            return nil
        },
    })
}
```

**2. How it works.**

During command building, `dispatch.BuildCommandsWithClient` checks the override map before creating a generic command. If your API method has an override, the override command is used as-is and no dispatch function is called.

**When to use this:**

- The command needs interactive behavior or special I/O
- The flags or output format diverge significantly from the standard pattern
- The command does not map to a single API call

**When NOT to use this:**

- You just need to call the SDK differently -- use a dispatch function instead
- You want different flag names -- update the mapping table instead

## How to Add a Top-Level Builtin Command

Some commands are not overrides of generated API methods at all -- they are new top-level commands that compose multiple API calls or provide high-level UX not representable in the dispatch pipeline. `thread-read`, `message-read`, `cache`, and `resolve` are examples.

**1. Write the Cobra command constructor.**

Create a file in `internal/override/` with a constructor function that accepts `*slack.Client`:

```go
package override

import (
    "fmt"

    "github.com/poconnor/slack-cli/internal/exitcode"
    "github.com/slack-go/slack"
    "github.com/spf13/cobra"
)

func newMyCmd(client *slack.Client) *cobra.Command {
    cmd := &cobra.Command{
        Use:   "my-cmd",
        Short: "Does something useful",
        RunE: func(cmd *cobra.Command, args []string) error {
            if client == nil {
                return formatAndExit(cmd, fmt.Errorf("SLACK_TOKEN is not set"), exitcode.AuthError)
            }
            // Your implementation
            return nil
        },
    }
    cmd.Flags().String("channel", "", "Channel ID")
    return cmd
}
```

**2. Register it in `RegisterBuiltins`.**

Add your constructor call in `internal/override/api_list.go`:

```go
func RegisterBuiltins(root *cobra.Command, client *slack.Client) {
    // ... existing registrations ...
    root.AddCommand(newMyCmd(client))
}
```

**3. Key conventions for builtin commands:**

- Always check `client == nil` first and return `exitcode.AuthError`
- Use `formatAndExit(cmd, err, code)` for all error returns -- it writes JSON to stderr and returns a typed exit error
- Use `ensureCacheReady(cmd, client)` if your command needs name resolution
- Use `MarkFlagsMutuallyExclusive` and `MarkFlagsRequiredTogether` for flag constraints -- let Cobra enforce them at the framework layer, not in RunE
- Accept `--url` OR `--channel`+`--ts` for commands that take a message reference (see `thread-read` and `message-read` as the canonical pattern)

**4. Shared helpers in `internal/override/`:**

| Helper | Purpose |
|--------|---------|
| `formatAndExit(cmd, err, code)` | Write JSON error to stderr, return exit error |
| `ensureCacheReady(cmd, client)` | Auto-warm or migrate cache if needed |
| `parseSlackURL(rawURL)` | Extract channel and timestamp from a Slack URL |
| `resolveChannelTSFromValues(url, channel, ts)` | Resolve channel+ts from --url or explicit flags |
| `resolveUser(userID, botID, idMap)` | Resolve user ID to display name with bot detection |
| `parseSlackTimestamp(ts)` | Convert Slack float timestamp string to time.Time |
| `formatMessages(msgs, asJSON, w)` | Format []readMessage as plain text or JSON |

## Testing

### Conventions

- **Standard library** `testing` package only, plus `github.com/google/go-cmp/cmp` for deep comparisons
- **No assertion libraries** (no testify, no gomega)
- **Table-driven tests** with `t.Run` subtests
- **Test file location**: Tests live next to the code they test (e.g., `builder_test.go` in `internal/dispatch/`)

### Running Tests

```bash
# All tests with race detection
make test

# A specific package
go test -race -count=1 ./internal/dispatch/...

# A specific test
go test -race -count=1 -run TestExecute ./internal/dispatch/...

# Verbose output
go test -race -count=1 -v ./internal/dispatch/...
```

### Writing a Test

Follow this pattern:

```go
func TestSomething(t *testing.T) {
    tests := []struct {
        name    string
        input   SomeInput
        want    SomeOutput
        wantErr string  // empty means no error expected
    }{
        {
            name:  "valid input",
            input: SomeInput{Field: "value"},
            want:  SomeOutput{Result: "expected"},
        },
        {
            name:    "missing required field",
            input:   SomeInput{},
            wantErr: "field is required",
        },
    }

    for _, tt := range tests {
        t.Run(tt.name, func(t *testing.T) {
            got, err := DoSomething(tt.input)

            if tt.wantErr != "" {
                if err == nil {
                    t.Fatalf("expected error containing %q, got nil", tt.wantErr)
                }
                if !strings.Contains(err.Error(), tt.wantErr) {
                    t.Fatalf("error %q does not contain %q", err.Error(), tt.wantErr)
                }
                return
            }
            if err != nil {
                t.Fatalf("unexpected error: %v", err)
            }

            if diff := cmp.Diff(tt.want, got); diff != "" {
                t.Errorf("mismatch (-want +got):\n%s", diff)
            }
        })
    }
}
```

### What to Test

- **Dispatch functions**: Test flag extraction, required flag validation, and the shape of the return value. You can test against the dispatch registry without a live Slack client by checking registration and error cases.
- **Validators**: Test boundary cases (`validate.ChannelID` with valid IDs, invalid IDs, empty strings).
- **Output formatting**: Test JSON output and pretty-print output against expected strings.
- **Pagination**: Test cursor loop behavior, early termination, context cancellation.

### End-to-End Tests

End-to-end tests require a real `SLACK_TOKEN` and are gated behind a build tag:

```bash
make e2e  # Builds the binary, then runs go test -tags e2e
```

These live in `e2e_test.go` at the project root.

## Code Style

### Linting

```bash
make lint  # Runs golangci-lint run ./...
```

Fix all lint issues before submitting a PR. If a lint rule conflicts with the correct approach for a specific case, suppress it with a `//nolint:rulename` comment and explain why.

### Naming

- **Dispatch functions**: `impl<Category><Action>` (e.g., `implConversationsList`, `implReactionsAdd`)
- **Flag names**: kebab-case matching the Slack API parameter (e.g., `--channel`, `--thread-ts`, `--exclude-archived`)
- **Packages**: Single, lowercase word (e.g., `dispatch`, `registry`, `validate`)
- **Test functions**: `Test<FunctionName>` or `Test<FunctionName>_<scenario>` (e.g., `TestExecute`, `TestExecute_unknownMethod`)

### Generated Files

Never edit files with `// Code generated by cmd/introspect; DO NOT EDIT.` at the top. If the generated output is wrong, fix the generator (`cmd/introspect/`) and re-run `make generate`.

## PR Process

### Workflow

1. **Branch** from `main` with a descriptive name (e.g., `feat/impl-reactions`, `fix/pagination-cursor`)
2. **Write tests first** (red), then implementation (green), then clean up (refactor). This is the BDD red-green-refactor cycle.
3. **Keep PRs under 300 lines.** If a change is larger, split it into smaller, independently reviewable pieces.
4. **Run checks locally** before pushing:
   ```bash
   make test && make lint
   ```
5. **All tests MUST pass** before the PR can be merged
6. **Reviews MUST complete** before merging -- no exceptions

### Commit Messages

Use conventional commit style. All commits must include both co-authors:

```
feat: implement reactions.add dispatch function

Adds the real dispatch implementation for reactions.add, replacing the
generated stub. Validates required flags (name, channel, ts) and calls
AddReactionContext on the SDK client.

Co-Authored-By: Peter O'Connor <poconnor@stackoverflow.com>
Co-Authored-By: Claude <noreply@anthropic.com>
```

### Secrets

Never commit tokens, credentials, or secrets. The `SLACK_TOKEN` is read from the environment at runtime. If you see a token in the codebase, remove it immediately and rotate the token.
